## Kelmy

### How I wrote prompts

When I wrote prompts I had to think of all of the conditions and outputs I wanted the AI to give me first. I first tried to give the AI context on what I was doing so it understands my situation better. Once I was done with giving it context, I started to describe a step-by-step process on what I wanted the AI to do. When describing what I wanted the AI to do, I tried to be as descriptive and detailed as possible, so that the AI does not go and do something else that I did not want it to do. 

### How I validated AI output

When it came to validating AI output, my approach would be seeing all of the files the AI has affected or created. Then, I would go into each file and review the code it has written or the changes it has made. I made sure that all of my requirements that I have stated in my prompt were met and that it did not change or create anything that I did not advise it to do. 

### What I learned about spec-driven development

I learned a lot about spec-driven development. Through this project, I learned that spec-driven development makes implementation faster and cleaner because clear requirements force me to define scope and data needs, which reduced rework and helped me build each feature in logical increments. Writing specs first also improved consistency across UI, hooks, services, and tests, since every change could be thought out first and how it all would work together in the future instead of making it all in the same time.


## What they would change next 

Next time, I would improve by making the specs more measurable up front, especially with explicit edge cases, non-functional requirements, and test scenarios. There were times where I would write a prompt that would later be altered or completely redone. I would make sure to plan ahead more often and truly reflect on what I am asking the AI to do. I would also start prompting the AI to make sure that everything that I am doing is not redundant or illogical.
